# Implementation notes тАФ Event Visuals

This document covers what was built for `CODING_TEST.md` and the decisions behind it.

## Running it

```bash
# 1. install + key + db
composer install && npm install
cp .env.example .env && php artisan key:generate
php artisan migrate

# 2. seed a sensible amount of data (the default is 1.25M rows тАФ override it)
SEED_ROWS=20000 php artisan db:seed

# 3. run (serves app, queue worker, scheduler, vite)
composer dev
```

Emails use the `log` mailer by default, so confirmations/reminders land in
`storage/logs/laravel.log`. They are **queued**, so a queue worker must run
(`composer dev` starts `queue:listen`). Reminders are driven by the scheduler
(`php artisan schedule:work`, also part of `composer dev`).

To exercise reminders immediately without waiting for the hourly schedule:

```bash
php artisan events:send-reminders
```

## The two pages (both at `/events-visual-1` and `/events-visual-2`)

Two deliberately distinct layouts over the same data/filter layer:

- **Visual 1 тАФ card grid.** Image-led responsive grid, hover motion, infinite
  scroll.
- **Visual 2 тАФ timeline / agenda.** Events grouped by day on a vertical
  timeline with sticky day headers.

Shared logic lives in `useEventBrowser.ts` (filtering, pagination, request
de-duplication) and the `components/events/*` components, so the two pages only
differ in presentation.

## Key decisions & trade-offs

### Built for the large dataset
The seeder produces up to 1.25M events. Choices that follow from that:
- **`simplePaginate`, not `paginate`** for browsing тАФ avoids a `COUNT(*)` over a
  filtered slice of a million-plus rows on every page request.
- **New indexes** (`add_indexes_to_events_table`): `created_time` (date filter +
  ordering) and `(latitude, longitude)` (city bounding-box filter).
- **Reminders are driven from the `attendees` table**, not the events table, so
  the nightly cost scales with how many people registered, not with 1.25M
  events.

### Images тАФ local, 2+ per event (`EventImages` + `public/images/events/*.svg`)
Events have no stored imagery. Rather than back-fill image rows for 1.25M
events, images are derived deterministically from the event category + a hash of
its id, drawn from a local pool of SVG placeholders (generated by
`scripts/gen-placeholder-images.mjs`). The pipeline is end-to-end (model accessor
тЖТ API тЖТ gallery UI); in a real app the same accessor would read from an uploads
table instead of the pool. Everything is served locally тАФ no external/hotlinked
URLs.

### Addresses тАФ offline reverse geocoding (`ReverseGeocoder` + `CityDirectory`)
Events only carry lat/lng. The dataset jitters each event by тЙд0.5┬░ around a known
set of city anchors, so a nearest-anchor lookup recovers a reliable
"City, Country" with **no external API** тАФ deterministic and free at any scale.
The same city list powers the location filter.

### Timezones
`created_time` is the event start as a UTC unix timestamp. Since events are
global, times are rendered in **the viewer's own timezone** via the browser
`Intl` API and labelled as local time, with a relative hint ("in 3 days"). Date
filtering interprets the chosen days as UTC boundaries. A fuller solution would
store each event's IANA timezone (derived from coordinates) to show true
event-local time; that was out of scope here and is noted as a follow-up.

### Filtering
Beyond the required **date** and **location** filters, the bar also supports
**category** and **status**. Location filtering uses an indexed bounding box
around the selected city anchor (┬▒0.6┬░, matching the dataset's jitter).

### Attendees & emails
- Public registration by name + email (`attendees` table, unique per
  event+email; `firstOrCreate` makes re-registration idempotent).
- A queued **confirmation** email on first registration.
- An idempotent `events:send-reminders` command (scheduled hourly) sends a
  **3-day** and a **24-hour** reminder, stamping `reminder_3d_sent_at` /
  `reminder_24h_sent_at` so each goes out at most once. The two windows are
  disjoint, so an event never triggers both at the same run.

## Notes / scope
- The original `Events/Index` baseline page and its `/events/data` endpoint are
  left intact (I only fixed a pre-existing `aplyFilters` typo on its button). The
  new work lives behind `/events/browse` and the two visual routes.
- Cancelled events remain visible in browsing (clearly badged), but their detail
  page disables registration and they are excluded from reminders.
